# Measurements start off; an explicit Start press, not app launch, begins capture

The app previously auto-started the capture pipeline the moment the window appeared (`AppShellView`'s `.task` called `AudioPipelineViewModel.start()`), matching the "opens and just works" philosophy -- no extra tap before the first reading. We changed this: the app now launches with capture stopped (the Stop/Start control reads "START"), and only an explicit press begins capture.

We made this change because auto-starting collides with a real, still-open bug: macOS shows its microphone permission prompt asynchronously the first time the engine actually touches the mic, and the app has no explicit permission request or retry logic -- `AudioPipelineViewModel.startCapture`'s `catch` silently swallows a failed `captureEngine.start()` and resets to `.stopped` with no user-facing error and no retry once permission is later granted. If the OS prompt hasn't been answered yet when auto-start fires at launch, capture fails silently and the waterfall just sits dead with no explanation. Tying the permission prompt to a deliberate Start press instead gives the tech a clear action-outcome relationship (press Start, see the permission dialog) and an obvious thing to retry if it fails, rather than a mysterious silent failure at app open. This is a workaround for the diagnosed bug, not a fix for it -- the underlying gap (no explicit `AVCaptureDevice.requestAccess`/retry-on-grant) is still open, noted in CLAUDE.md's Architecture section.

**Addendum (underlying bug fixed).** `AudioPipelineViewModel.performStartCapture` now explicitly requests mic permission (`AVAudioApplication.requestRecordPermission()`) and awaits the user's answer *before* touching the engine — the engine can no longer start against an unanswered prompt, so the silent-dead-capture race this ADR worked around is closed. A denial lands in the `.stopped` state and shows a red "MIC ACCESS DENIED" indicator in the Input Device plate (same treatment as DISCONNECTED; priority DISCONNECTED > MIC ACCESS DENIED > CAPTURE UNAVAILABLE) — cleared automatically the next time a Start attempt finds permission granted, so granting access in System Settings and pressing Start again is the visible recovery path. Revoking permission *mid-capture* is also handled (found by user report): macOS doesn't stop an already-granted process's engine — coreaudiod keeps delivering exact digital zeros, so hops flow and the start-time gate never re-runs (symptom: a live-looking "6 Hz / -198 dB" readout of pure silence). `AudioPipelineViewModel.receive` counts consecutive exact-silence hops (below -180 dB, comfortably under anything a real ADC produces) and after ~1s asks TCC once; if the status reports non-granted, capture stops and the indicator shows. Exact silence with permission still granted (a loopback device with nothing routed, e.g. BlackHole) deliberately does not trigger anything. The measurements-start-off behavior itself is kept: it remains good product behavior (the permission prompt appears in response to a deliberate Start press, a clear action-outcome relationship), it just no longer carries the weight of being a bug workaround.

Device enumeration itself still happens unconditionally at `AudioPipelineViewModel.init()` (matching how `SignalGeneratorEngine` already populates its Output Device list regardless of on/off state), so the Input Device picker is populated immediately at launch -- only the actual AVAudioEngine capture start is gated behind the user's Start press. `resumeCapture()` (the Start button's action) falls back to the same system-default/persisted-choice resolution `start()` used, so a cold launch's first Start press behaves identically to the old auto-start, just user-triggered instead of automatic.
