# Octave-banding resolution is one shared setting for RTA and waterfall, not independent per view

The RTA's bar count used to be a fixed constant (`RTABinning.defaultBarCount = 48`, a judgment call with no user control). The user asked for it to become selectable, expressed in the vocabulary real analyzers use for band density -- "1/1" through "1/48 octave" (bars-per-octave 1 through 48) -- via `RTABandingResolution`. They then asked for the same banding to apply to the waterfall too.

We considered giving the waterfall its own independent resolution control, mirroring how Time Averaging is already scoped per-view in this app (CONTEXT.md "Time Averaging": "meant to be selected per-view eventually"). We asked the user directly and they chose one shared setting instead: a single `AudioPipelineViewModel.bandingResolution`, with one picker (in the graph zone, next to the Waterfall/RTA toggle) visible regardless of which view is showing, rather than two independent controls. The reasoning: "how much frequency detail" reads as one property of the analysis itself, not two separate per-view preferences -- unlike Time Averaging, which genuinely can make sense to want faster on one view and slower on another.

This meant the RTA's continuous peak-tracking pass (computed every hop regardless of which view is visible, so Peak stays indefinite per CONTEXT.md) and the waterfall's texture-write path both read the same `bandingResolution` value, and switching it clears the RTA bars' stale held peaks (a bar index means a different frequency once the bar count changes). The waterfall doesn't resample to a smaller texture -- it pre-bins magnitudes into the selected bands and expands each band's peak back out across its own bin range before writing to the GPU texture (`RTABinning.steppedMagnitudes`), so none of the Metal shader's per-pixel log-frequency remap needed to change.
